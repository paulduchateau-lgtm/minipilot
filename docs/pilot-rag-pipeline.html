Supprimer les donnees brutes des prompts IA

Pilot analyse des donnees tabulaires (Excel, CSV). Aujourd'hui, les rows sont injectees en brut dans les prompts. Ce n'est ni un probleme de RAG (les donnees ne sont pas du texte non structure), ni un probleme de recherche semantique. C'est un probleme de materialisation : pre-calculer l'intelligence a la transformation, pour que le LLM raisonne sur des faits derives, pas sur des lignes CSV.

⚠ Decision architecturale : Materialized Analytics, pas RAG classique

Un pipeline RAG classique (chunk → embed → cosine search) est concu pour du texte non structure (docs, PDF, mails). Les donnees de Pilot sont structurees et tabulaires. La bonne abstraction est une couche de materialisation : pre-calculer des vues analytiques (agregats, distributions, anomalies) au moment du transform, les stocker, et ne fournir au LLM que ces faits derives.

Upload & Transform Inchange

Le pipeline actuel reste identique. Les donnees arrivent dans clean_data avec types detectes et formats normalises.

2
Materialisation des vues analytiques Coeur V2

A la fin du transform, un moteur de materialisation genere 4 artefacts a partir des donnees brutes. Ces artefacts sont stockes en DB et ne changent que quand les donnees changent.

A. Schema FingerprintNom de table, colonnes, types, cardinalites.
"Table sinistres: 4538 rows, 32 cols. organisme (12 unique), cotisation (num, 45-380)..."
B. Statistical ProfileDistributions, outliers, correlations cles.
"Cotisation moy 156€ (σ=42). Bimodale: pic a 120€ (petits contrats) et 280€ (entreprises)."
↓
C. Dimension SummariesAgregats par dimension metier (groupby sur colonnes categoriques).
"MGEN: 245 benef, sinistralite 62%. AG2R: 180 benef, sinistralite 71%."
D. Anomaly SignalsEcarts significatifs, tendances, valeurs extremes.
"Alerte: organisme XYZ a une sinistralite de 142% (+2.3σ vs moyenne)."
-- Table : vues analytiques materialisees CREATE TABLE IF NOT EXISTS materialized_views ( id TEXT PRIMARY KEY, workspace_id TEXT NOT NULL REFERENCES workspaces(id), table_name TEXT NOT NULL, view_type TEXT NOT NULL, -- 'schema' | 'stats' | 'dimensions' | 'anomalies' content_text TEXT NOT NULL, -- resume en langage naturel content_json TEXT, -- donnees structurees pour le frontend computed_at DATETIME DEFAULT CURRENT_TIMESTAMP ); CREATE INDEX idx_mv_ws ON materialized_views(workspace_id, view_type);
Pourquoi materialiser plutot qu'embedder ?

Embeddings = utile quand la question est imprevue et qu'il faut chercher dans du texte libre. Ici, les questions sont previsibles ("suggere des rapports", "genere un rapport sur X") et les donnees sont structurees. Une materialisation deterministe est plus rapide (0 appel API), moins chere (0 token d'embedding), plus fiable (pas de perte d'info cosine), et plus auditable (on sait exactement ce que le LLM voit).

Si un besoin de recherche semantique emerge plus tard (ex: chat libre sur les donnees), on pourra ajouter des embeddings sur les content_text des vues materialisees — pas sur les rows brutes.

Statistiques descriptives avancees calculees cote serveur. Inclut distributions, quartiles, correlations entre colonnes numeriques, detection de bimodalite.

// Exemple Cotisation : moy=156€, med=142€, Q1=98, Q3=210, σ=42. Distribution bimodale : pic 1 a 120€ (petits contrats N=1,200), pic 2 a 280€ (entreprises N=890). Correlation : cotisation ↔ nb_beneficiaires r=0.74 (forte). Taux de remplissage : 98.2% (82 valeurs nulles sur 4,538).

Agregations automatiques par chaque colonne categorique (GROUP BY). Detect la colonne de regroupement la plus pertinente (plus haute cardinalite utile). Genere un resume par modalite.

Anomaly Signals Zero IA

Detection statistique d'anomalies : z-scores > 2, valeurs hors IQR, tendances monotones, ruptures de distribution. Chaque anomalie est textualisee.

// Exemple 3 anomalies detectees : [ALERTE] Organisme "Previfrance" : sinistralite 142% (z=+2.3, moyenne 68%). Investiguer. [TENDANCE] Cotisation moyenne en hausse de 8% sur les 6 derniers mois. [OUTLIER] 23 lignes avec montant_sinistre = 0 et statut "rembourse". Incohérence possible.

Phase 4 — Generation & Streaming
8
Streaming SSE + Code Interpreter pattern Frontend + Backend

Le LLM genere une spec de rapport (titres, types de charts, KPIs a calculer), pas les donnees du rapport. Les donnees sont calculees cote serveur a partir des clean_data via des requetes SQL generees par le LLM.

LLM OutputSpec : "Barchart cotisation par organisme,
KPI: sinistralite moyenne,
Filter: type_contrat = collectif"
→
Query BuilderTraduit la spec en queries SQL
sur clean_data (cote serveur,
jamais envoye au LLM)
→
Rapport finalSpec + donnees calculees
→ JSON complet pour le frontend
Pourquoi ce pattern "spec + compute" ?

Aujourd'hui : le LLM invente les chiffres du rapport a partir de l'echantillon de 20 rows. Les valeurs sont souvent fausses ou approximatives. Demain : le LLM genere une specification (quoi calculer, comment grouper, quels filtres), et le serveur execute les calculs reels sur 100% des donnees. Les chiffres sont exacts.

9
Frontend : affichage progressif Frontend

Les tokens SSE arrivent en <2s. Le frontend affiche d'abord le titre et les KPIs, puis les sections au fur et a mesure. Fin du spinner de 30 secondes.

2. Plan d'implementation

Phase 1 — Materialized Views (impact immediat)Backend

Creer la table materialized_views. Implementer les 4 materialiseurs (schema, stats, dimensions, anomalies) dans une fonction materialize(workspaceId). L'appeler a la fin de POST /transform. Remplacer buildDataSummary() et buildColumnStats() par une lecture de materialized_views. Le prompt passe de ~12K a ~2K tokens. Zero appel IA supplementaire — 100% deterministe.

Phase 2 — Refactoring des promptsAI

Remplacer le contenu des prompts suggest-reports et generate-report. Utiliser les content_text des vues materialisees. Adapter les system prompts pour que le LLM raisonne sur des faits derives plutot que sur des echantillons de rows. Valider la qualite des suggestions.

Phase 3 — Streaming SSEFrontend + Backend

Ajouter le streaming SSE sur les routes IA. Backend : res.writeHead(200, {'Content-Type': 'text/event-stream'}), pipe les tokens Claude/Mistral au client. Frontend : fetch + ReadableStream avec affichage incremental du JSON. Time to first token : <2s.

Phase 4 — Spec + Compute (qualite des chiffres)Architecture

Refactorer generate-report : le LLM genere une spec (colonnes, groupby, filtres, type de chart) et le serveur execute les requetes SQL pour produire les vraies donnees du rapport. Plus de chiffres inventes. Necessite un QueryBuilder qui traduit les specs LLM en requetes SQL sur clean_data.

Phase 5 — Embeddings optionnels (chat libre)Optionnel

Si le chat libre sur les donnees devient un besoin, embedder les content_text des vues materialisees (pas les rows brutes). Vector search pour trouver les vues pertinentes a une question. Reutiliser le pipeline Voyage AI de Sailor. Mais ce n'est pas prioritaire — les phases 1-4 couvrent 90% du besoin.
