Pipeline Zero Data in Prompts

Architecture cible pour que le LLM ne voie jamais de donnees reelles. Il agit comme un planificateur d'analyses, un moteur local execute les calculs, puis le LLM narre les resultats agreges. Aucune donnee individuelle ne quitte le serveur.

1. Probleme actuel

Aujourd'hui, le serveur injecte des donnees brutes dans chaque prompt LLM. Voici ce qui transite vers les API cloud (Anthropic, Mistral) :

⚠ Architecture actuelle
✗ Jusqu'a 200 lignes brutes envoyees dans le prompt
✗ Echantillons de donnees reelles (montants, identifiants, dates)
✗ Distributions completes (toutes les valeurs uniques + frequences)
✗ Statistiques min/max/avg par colonne
✗ Cross-tabulations categoriel x numerique
✗ Contexte projet (nom, secteur, objectifs)
✗ Historique de chat complet
🔒 Architecture cible
✓ Noms de tables, colonnes, types (schema uniquement)
✓ Nombre de lignes, cardinalite par colonne
✓ Contexte projet (non-sensible)
✓ Resultats agreges uniquement (totaux, pourcentages)
✓ Seuil de k-anonymat (groupes < 5 → masques)

Jamais : lignes individuelles, identifiants, montants unitaires, dates, noms, distributions brutes

2. Pipeline securise en 3 phases

Phase 1 — Schema Analysis
1
Extraction du schema (zero donnees) Safe

Le serveur extrait uniquement les metadonnees structurelles. Aucune valeur reelle.

SQLSELECT DISTINCT table_name FROM clean_data WHERE workspace_id = ?
SELECT columns FROM clean_data WHERE table_name = ? LIMIT 1
SELECT COUNT(*) FROM clean_data WHERE table_name = ?
↓
buildSecureSchema() Produit : table_name, column_name, column_type, row_count, cardinality (nombre de valeurs uniques)
Pas de valeurs, pas de min/max, pas d'echantillons
// Ce qui sort de buildSecureSchema() : [{ "name": "prestations", "rowCount": 1280, "columns": [ { "name": "montant_prestation", "type": "number", "cardinality": 847 }, { "name": "garantie", "type": "string", "cardinality": 12 }, { "name": "date_sinistre", "type": "date", "cardinality": 365 } ]}] // Aucune valeur reelle, aucun echantillon, aucune statistique
2
Prompt LLM → Analysis Plan Cloud

Le LLM recoit le schema + le contexte projet et retourne un plan d'analyse structure (JSON).

Prompt envoye au LLM Schema des tables (noms, types, cardinalite)
Contexte projet (nom, secteur, objectifs)
Consigne : "Conçois un plan d'analyse pour ce jeu de donnees"
Zero donnee reelle dans ce prompt
→
LLM retourne Analysis Plan JSON :
• KPIs a calculer (quelle aggregation, quelle colonne)
• Sections : type de chart, group_by, metrics, filtres
• Configs Recharts (xKey, yKeys, colors)
// Extrait de l'Analysis Plan retourne par le LLM : { "kpis": [ { "label": "Total prestations", "table": "prestations", "aggregation": "sum", "column": "montant_prestation", "format": "money" } ], "sections": [ { "title": "Prestations par garantie", "chart_type": "bar", "table": "prestations", "group_by": "garantie", "metrics": [{ "column": "montant_prestation", "aggregation": "sum" }], "sort": { "by": "total", "order": "desc" }, "limit": 10 } ] } // Le LLM decrit QUOI calculer, pas les resultats
Phase 2 — Execution locale
3
Validation du plan Local

Chaque reference du plan est validee contre le schema reel. Deny-by-default.

PlanValidator Whitelist de tables → extraite de clean_data
Whitelist de colonnes → par table, depuis le JSON columns
Whitelist d'aggregations → count | count_distinct | sum | avg | min | max | median | ratio
Whitelist de chart_type → bar | grouped_bar | area_multi | composed | pie_multi | table

Si un nom de table/colonne/fonction n'est pas dans la whitelist → plan rejete, retry ou fallback template
4
Execution SQL locale Local

Le PlanExecutor traduit le plan en requetes SQL parametrees et les execute contre SQLite. Zero appel reseau.

Plan → SQL Le plan est traduit en requetes SELECT avec json_extract(row_data, '$.col'), GROUP BY, ORDER BY, LIMIT
Les noms de colonnes sont valides (whitelist) puis injectes dans le path JSON.
Les valeurs de filtre utilisent des prepared statements (?).
↓
SQL genere (exemple)SELECT json_extract(row_data, '$.garantie') AS garantie, SUM(CAST(json_extract(row_data, '$.montant_prestation') AS REAL)) AS total FROM clean_data WHERE table_name = ? AND workspace_id = ? AND json_extract(row_data, '$.montant_prestation') IS NOT NULL GROUP BY json_extract(row_data, '$.garantie') ORDER BY total DESC LIMIT ?
↓
Resultat : donnees chart-ready [{garantie: "Optique", total: 890000}, {garantie: "Dentaire", total: 650000}, ...]
+ KPIs calcules : {label: "Total", value: "2 450 000 EUR"}
+ Configs Recharts (xKey, yKeys, colors) issues du plan
Phase 3 — Narration
5
Sanitisation des resultats Local

Avant d'envoyer les resultats au LLM, le InsightSanitizer filtre ce qui peut sortir.

Regles de filtrage Autorise : labels de categories (group_by), totaux, moyennes, pourcentages, tendances
Interdit : lignes individuelles, identifiants, dates unitaires, montants par personne
K-anonymat : groupes < 5 lignes → fusionnes dans "Autres" ou supprimes du prompt
Arrondi : valeurs monetaires arrondies au millier pour eviter la re-identification
6
Generation des insights Cloud

Le LLM recoit les resultats sanitises et genere le texte analytique.

// Prompt Phase 3 (envoye au LLM) SCHEMA (rappel) : prestations (garantie, montant_prestation, ...) RESULTATS AGREGES : KPI "Total prestations" : 2 450 000 EUR KPI "Beneficiaires" : 1 234 Section "Prestations par garantie" (bar) : Optique: 890k | Dentaire: 650k | Hospitalisation: 430k | ... MISSION : Genere un insight analytique professionnel pour chaque section. Retourne : { "insights": [{"section_id", "text"}] }
Assemblage
7
Assemblage du rapport final Local

Le ReportAssembler fusionne les donnees (Phase 2) et les textes (Phase 3) dans le format rapport standard.

+

5. Cas limites et securite

Plan invalide du LLM

Le LLM peut generer un plan referencant des colonnes inexistantes.

1. validatePlan() retourne {valid: false, errors: [...]}
2. Retry avec prompt correctif : "La colonne X n'existe pas, les colonnes disponibles sont Y, Z"
3. Si 2e echec → fallback sur un template generique (count par categoriel, sum/avg par numerique)

Prevention d'injection SQL

Le LLM ne genere jamais de SQL. Il genere un plan JSON declaratif. Le PlanExecutor construit le SQL en interne :

• Noms de tables/colonnes : valides par whitelist (pas d'echappement — rejet pur)
• Valeurs de filtre : prepared statements (?)
• Fonctions d'aggregation : set fixe hardcode
• Path json_extract : construit depuis '$.' + whitelist_column (alphanum + underscore only)

K-anonymat (seuil de groupe)

Si un GROUP BY produit un groupe de moins de N lignes (defaut: 5), le InsightSanitizer fusionne ce groupe dans "Autres" avant envoi au LLM Phase 3. Empeche la re-identification par petits groupes.

Complexite du plan

Limites hardcodees : max 6 KPIs, max 8 sections, max 3 metrics par section, max 20 resultats par group_by, max 30 requetes SQL par plan.

Defense en profondeur (mode Ollama)

Meme en mode local (Ollama), le pipeline securise s'applique. Si Ollama est un jour configure pour proxyer vers un endpoint distant, ou si une future version ajoute de la telemetrie, les donnees restent protegees. La securite ne depend pas de la couche transport.

6. Sequencage d'implementation

A
schema-builder.js + plan-prompt.js

Extraction du schema securise + templates de prompts. Testable independamment.

B
plan-validator.js

Validation whitelist avec tests exhaustifs (colonnes invalides, injections, edge cases).

C
plan-executor.js

Traduction plan → SQL + execution. Piece la plus complexe. Tester avec clean_data JSON blobs.

D
insight-sanitizer.js + report-assembler.js

Filtrage k-anonymat + fusion donnees/texte en rapport JSON standard.

E
Integration dans index.js

Remplacement des 6 endpoints AI par le pipeline 3 phases. Flag AI_SECURE=true pour rollout progressif.

F
Adaptation du chat

Pipeline securise pour le chat conversationnel. Cache de "faits calcules" par session.
