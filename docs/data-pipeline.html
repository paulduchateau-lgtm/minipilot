Phase 2 — Intelligence
5
Construction du contexte IA Backend

Avant chaque appel LLM, le serveur execute des requetes SQL pour construire un contexte de donnees riche. Il n'y a pas de text-to-SQL : le LLM ne genere ni n'execute jamais de SQL. Toutes les donnees sont pre-extraites cote serveur.

buildDataSummary(n) Requetes SQL executees :
SELECT DISTINCT table_name FROM clean_data WHERE workspace_id = ? SELECT columns FROM clean_data WHERE table_name = ? LIMIT 1 SELECT COUNT(*) FROM clean_data WHERE table_name = ? SELECT row_data FROM clean_data WHERE table_name = ? LIMIT {n} // n = 5 pour suggestions, 20 pour generation, 10 pour chat
Produit : nom de table, colonnes + types, rowCount, echantillon de n lignes
buildColumnStats() Requetes SQL executees :
SELECT row_data FROM clean_data WHERE table_name = ? AND workspace_id = ? // Puis calcul JS sur toutes les lignes : // - Numeriques: Math.min(), Math.max(), reduce(sum)/count // - Textuelles: new Set(values).size, [...set].slice(0,5)
Produit : pour chaque colonne → {type, nullCount, min, max, avg} ou {uniqueCount, sampleValues}
buildFullDataContext() — Chat uniquement Charge toutes les lignes + calcule distributions categorielle, cross-tabulations, percentiles.

2. Mode Premium vs Mode Offline

Le mode se bascule en runtime via POST /api/ai/mode avec {"mode": "local"} ou {"mode": "premium"}. Le pipeline data (etapes 1-4, 7-8) est identique. Seule la phase Intelligence (etapes 5-6) change.

6
Pas de fallback : Si Ollama est down → erreur 500
Prerequis : ollama pull ministral:3b

Point cle : le LLM ne genere pas de SQL

Le pipeline ne contient aucune phase text-to-SQL. Le LLM n'a pas acces a la base de donnees. Le serveur Node.js execute lui-meme les requetes SQL via better-sqlite3 (appels synchrones, in-process), construit un resume textuel des donnees, et l'injecte dans le prompt. Le LLM repond uniquement avec un JSON structure (rapport, KPIs, sections).

Les donnees qui apparaissent dans les charts sont calculees par le LLM a partir des echantillons et statistiques fournis dans le prompt. Pour les rapports complexes, le serveur envoie jusqu'a 20 lignes d'echantillon + les stats completes, ce qui permet au LLM de produire des agregations realistes.

4. Schema de base de donnees

SQLite en mode WAL avec foreign keys. Chaque table est scopee par workspace_id.

-- Espaces de travail isoles CREATE TABLE workspaces ( id TEXT PRIMARY KEY, slug TEXT UNIQUE NOT NULL, name TEXT NOT NULL, industry TEXT ); -- Fichiers importes (donnees brutes) CREATE TABLE uploaded_files ( id TEXT PRIMARY KEY, name TEXT, type TEXT, -- "xlsx" | "csv" | "json" columns TEXT, -- JSON array row_count INTEGER, workspace_id TEXT REFERENCES workspaces(id) ); -- Lignes brutes (JSON par ligne) CREATE TABLE data_rows ( id INTEGER PRIMARY KEY AUTOINCREMENT, file_id TEXT REFERENCES uploaded_files(id), sheet_name TEXT, row_data TEXT -- JSON object ); -- Donnees nettoyees et typees CREATE TABLE clean_data ( id INTEGER PRIMARY KEY AUTOINCREMENT, table_name TEXT, columns TEXT, -- JSON: [{name, type, nullable}] row_data TEXT, -- JSON: {col1: val1, ...} types convertis workspace_id TEXT REFERENCES workspaces(id) ); -- Rapports generes par IA CREATE TABLE reports ( id TEXT PRIMARY KEY, title TEXT, subtitle TEXT, objective TEXT, color TEXT, icon TEXT DEFAULT 'BarChart3', kpis TEXT, -- JSON: [{label, value, trend, bad}] sections TEXT, -- JSON: [{title, type, insight, data, config}] shared INTEGER DEFAULT 0, starred INTEGER DEFAULT 0, trashed INTEGER DEFAULT 0, source TEXT DEFAULT 'ai', workspace_id TEXT REFERENCES workspaces(id) ); -- Contexte projet + Sessions chat + Logs CREATE TABLE project_context (...); CREATE TABLE chat_sessions (... messages TEXT DEFAULT '[]' -- JSON multi-tour); CREATE TABLE usage_logs (... action TEXT, query TEXT, themes TEXT);
